# 私人定制小程序简单版开发方案

## 1. 项目定位

本项目第一版定位为一个轻量可演示版本，用于展示高端私人服装定制小程序的核心体验：收集用户身材、审美、版型和穿着场景偏好，并基于 mock 数据生成推荐方案，最后完成一条模拟下单流程。

第一版不追求真实业务闭环，重点是让用户、客户或团队可以直观看到产品逻辑和交互体验。

服务基础信息：

- 服务名称：高端私人定制
- 服务定位：高端私人服装定制
- 目标客户：高净值客户
- 主打风格：老钱风、轻奢休闲风
- 量体安排：线下量体门店

## 2. 开发目标

- 做出一个可点击、可演示的简单版本。
- 使用本地 mock 数据，不接真实后端。
- 保留核心交互体验，包括填写、选择、推荐、定制、确认。
- 不做复杂算法和真实 AI，只用规则或固定数据模拟推荐结果。
- 不接真实支付、不做真实订单系统。
- 页面风格保持高级、克制，突出老钱风与轻奢休闲风。
- 在关键节点加入服务保障和线下量体安排，增强下单信任感。

## 3. 推荐技术方案

### 方案 A：微信小程序原生

适合直接部署到微信小程序。

优点：

- 可直接在微信开发者工具中开发、预览、上传。
- 页面结构符合微信小程序发布要求。
- 后续接入微信登录、支付、客服、订阅消息更顺。
- 适合从简单版继续迭代到正式上线版。

缺点：

- 开发速度比纯 HTML 原型略慢。
- 页面、样式、组件需要按小程序规范处理。

### 方案 B：uni-app / Taro

适合未来希望同时发布到 H5、App 或其他小程序平台。

优点：

- 一套代码可适配多端。
- 生态成熟，组件和工程化能力更强。
- 后续如果要做 H5 官网或 App，可减少重复开发。

缺点：

- 对简单第一版来说工程复杂度略高。
- 需要额外关注跨端兼容问题。

### 方案 C：HTML + CSS + JavaScript

只适合做原型参考，不建议作为最终微信小程序部署方案。

优点：

- 开发速度最快。
- 浏览器即可打开。
- 适合前期确认视觉和流程。

缺点：

- 不能作为普通小程序页面直接发布。
- 如果通过 `web-view` 承载 H5，需要配置业务域名，体验和能力都不如原生小程序。
- 后续仍需要迁移到小程序原生或跨端框架。

### 推荐选择

既然目标是部署到微信小程序，第一版建议采用 **微信小程序原生** 实现。现有 HTML 原型只作为视觉和流程参考，不作为最终部署代码。

第一版不接真实后端，可使用小程序本地页面状态和 mock 数据完成演示；后续上线前再逐步接入微信登录、云开发或自建后端、微信支付和订单系统。

## 4. 功能范围

### 4.1 首页

首页用于建立产品第一印象，并引导用户开始定制。

功能包括：

- 服务名称：高端私人定制。
- 服务定位展示：高端私人服装定制，服务高净值客户。
- 主打风格展示：老钱风、轻奢休闲风。
- 主按钮：开始定制。
- 快捷入口：身材档案、风格测试、我的方案、预约顾问。
- 展示一张 mock 推荐方案卡片。
- 展示一张品牌服务卡，说明面料甄选、版型打磨、工艺细节和一对一专属定制。

### 4.2 身材档案页

用于收集用户基础身材数据。

字段包括：

- 身高
- 体重
- 肩宽
- 胸围
- 腰围
- 臀围
- 袖长
- 裤长

身型选择：

- 直筒
- 梨形
- 倒三角
- 苹果型
- 沙漏型

交互要求：

- 支持输入 mock 尺寸。
- 支持点击选择身型。
- 点击保存后进入风格测试页。

### 4.3 风格测试页

用于收集用户审美、穿着场景和版型偏好。

可选内容包括：

- 风格：老钱风、轻奢休闲风、极简高级、正式晚宴。
- 颜色：黑色、灰色、藏蓝、驼色、酒红、橄榄绿。
- 版型：修身、合体、宽松、廓形。
- 场景：通勤、会议、约会、婚礼、旅行、日常。

交互要求：

- 风格、颜色、场景支持多选。
- 版型支持单选。
- 点击生成推荐方案后进入推荐方案页。

### 4.4 AI 推荐方案页

第一版不接真实 AI，只使用 mock 推荐结果。

页面展示 3 套方案：

- 老钱风商务套装
- 轻奢休闲通勤款
- 高级度假外套

每套方案展示：

- 方案名称
- 推荐理由
- 匹配度
- 预计价格
- 适合场景

交互要求：

- 点击某个方案后可进入定制工作台。
- 方案内容可以基于用户偏好做简单变化，也可以第一版固定展示。

### 4.5 定制工作台

用于模拟用户选择颜色、毛料、衣型、大小和工艺细节。

可选项包括：

- 衣型：大衣、西装、衬衫、针织、裤装。
- 颜色：深石墨黑、藏蓝、驼色、酒红、橄榄绿。
- 面料：精纺羊毛、羊绒混纺、粗花呢、亚麻混纺。
- 尺码/版型：修身、合体、宽松、廓形。
- 细节：暖金纽扣、姓名刺绣、半里布、斜插口袋。

交互要求：

- 点击选项后更新选中状态。
- 根据面料和细节简单计算 mock 价格。
- 点击加入方案后进入订单确认页。

### 4.6 订单确认页

用于汇总用户选择，并模拟购买动作。

页面展示：

- 选择的衣型
- 选择的颜色
- 选择的面料
- 选择的工艺细节
- 身材档案摘要
- 价格明细
- 预计工期
- 顾问复核提示
- 服务类型：私人定制工作室
- 服务标准：面料甄选、版型打磨、精细化工艺、一对一专属定制
- 量体安排：线下量体门店

交互要求：

- 点击模拟下单后显示成功弹窗。
- 点击预约顾问后显示预约成功提示。

## 5. 不做范围

第一版明确不做以下内容：

- 微信登录授权
- 真实用户系统
- 真实数据库
- 真实支付
- 真实订单状态流转
- 后台管理系统
- 真实 AI 推荐算法
- 图片上传识别身材
- 库存管理
- 优惠券和会员体系
- 物流系统

## 7. 页面流转

推荐用户路径如下：

```text
首页
  -> 身材档案页
  -> 风格测试页
  -> AI 推荐方案页
  -> 定制工作台
  -> 订单确认页
  -> 模拟下单成功
```

辅助路径：

```text
首页
  -> 我的方案
  -> 定制工作台
```

```text
订单确认页
  -> 预约顾问成功提示
```

## 8. 交互细节

第一版需要保留的交互：

- 页面跳转
- 表单输入
- 单选
- 多选
- 色卡选择
- 面料选择
- 工艺细节选择
- 价格实时变化
- 推荐方案切换
- 加入方案
- 模拟下单成功弹窗
- 预约顾问成功提示

交互复杂度控制：

- 不做复杂动画。
- 不做拖拽。
- 不做 3D 服装预览。
- 不做真实图片上传。
- 不做复杂表单校验。

## 9. 视觉方向

整体视觉应偏高端定制服装，而不是普通电商商城。

建议设计风格：

- 主色：深石墨黑
- 强调色：暖金色
- 背景色：米白或浅灰
- 组件风格：克制、清晰、有留白
- 图片风格：服装剪影、面料纹理、定制工艺细节
- 文案气质：低调、高级、专属，避免促销化表达
- 首页和订单页承担品牌信任背书，风格测试和推荐页承担定制转化

避免：

- 颜色过多
- 促销感太强
- 页面堆满商品
- 图标和按钮过度花哨

## 10. 项目结构建议

推荐使用微信小程序原生结构：

```text
miniprogram/
  app.js
  app.json
  app.wxss
  data/
    mock-data.js
  pages/
    home/
      home.wxml
      home.wxss
      home.js
      home.json
    profile/
      profile.wxml
      profile.wxss
      profile.js
      profile.json
    style-test/
      style-test.wxml
      style-test.wxss
      style-test.js
      style-test.json
    recommendations/
      recommendations.wxml
      recommendations.wxss
      recommendations.js
      recommendations.json
    customizer/
      customizer.wxml
      customizer.wxss
      customizer.js
      customizer.json
    order-confirm/
      order-confirm.wxml
      order-confirm.wxss
      order-confirm.js
      order-confirm.json
  assets/
project.config.json
```

文件职责：

- `app.json`：声明页面路径、窗口样式、底部 tab 等小程序全局配置。
- `app.wxss`：全局视觉变量、通用按钮、卡片、标签、色卡样式。
- `data/mock-data.js`：品牌信息、身材数据、偏好数据、推荐方案、定制选项。
- `pages/home/`：首页和品牌服务卡。
- `pages/profile/`：身材档案页。
- `pages/style-test/`：审美和场景偏好页。
- `pages/recommendations/`：AI mock 推荐方案页。
- `pages/customizer/`：定制工作台。
- `pages/order-confirm/`：订单确认、顾问复核和量体安排。
- `assets/`：服装示意图、面料纹理、图标等静态资源。
- `project.config.json`：微信开发者工具项目配置。

如未来需要多端发布，可改用 uni-app / Taro，页面模块仍按上述业务结构拆分。

不建议第一版使用 `web-view` 承载 HTML 原型，除非只是内部演示。原因是 `web-view` 需要配置业务域名，且小程序原生能力、导航体验、审核表达和性能都不如原生页面直接可靠。

## 10.1 微信小程序部署准备

上线前需要准备：

- 注册微信小程序账号。
- 获取小程序 AppID。
- 安装微信开发者工具。
- 在微信公众平台配置小程序基本信息、服务类目、头像、名称和简介。
- 使用微信开发者工具创建项目，并选择小程序原生项目。
- 使用真机预览检查页面、滚动、按钮点击和表单输入。
- 上传体验版，邀请内部成员测试。
- 提交微信审核。
- 审核通过后发布正式版。

第一版如果只是 mock 演示，不需要服务器域名；如果后续接入接口、支付、登录态或 H5 web-view，则需要补充服务器域名、业务域名、隐私协议和相关资质配置。

## 11. 开发步骤

### 阶段 1：项目搭建

- 使用微信开发者工具创建小程序项目。
- 配置 `app.json` 页面路由。
- 创建各页面的 `wxml`、`wxss`、`js`、`json` 文件。
- 创建 `app.wxss` 全局样式。
- 创建 mock 数据文件。
- 确认移动端预览尺寸。

### 阶段 2：首页开发

- 实现品牌头部。
- 展示服务名称、目标客户和核心转化信息。
- 实现开始定制按钮。
- 实现快捷入口。
- 实现推荐方案预览卡片。
- 实现品牌服务卡，展示高品质选材、精细化工艺、一对一专属定制。

### 阶段 3：身材档案页开发

- 实现基础信息输入。
- 实现尺寸字段输入。
- 实现身型选择。
- 实现保存后跳转。

### 阶段 4：风格测试页开发

- 实现老钱风、轻奢休闲风、极简高级、正式晚宴等风格多选。
- 实现颜色多选。
- 实现版型单选。
- 实现场景多选。
- 实现生成推荐按钮。

### 阶段 5：推荐方案页开发

- 展示 3 套 mock 推荐方案：老钱风商务套装、轻奢休闲通勤款、高级度假外套。
- 展示匹配度、价格和推荐理由。
- 实现选择方案进入定制工作台。

### 阶段 6：定制工作台开发

- 实现衣型选择。
- 实现颜色选择。
- 实现面料选择。
- 实现工艺细节选择。
- 实现价格实时计算。
- 实现加入方案。

### 阶段 7：订单确认页开发

- 汇总用户选择结果。
- 展示身材档案摘要。
- 展示价格明细。
- 展示预计工期。
- 展示服务类型：私人定制工作室。
- 展示线下量体安排：线下量体门店。
- 展示顾问复核和服务保障说明。
- 实现模拟下单成功弹窗。
- 实现预约顾问提示。

### 阶段 8：体验优化

- 检查移动端布局。
- 检查按钮点击反馈。
- 检查页面跳转是否顺畅。
- 检查空状态和成功状态。
- 统一文字、间距、颜色和按钮样式。

## 12. 验收标准

第一版完成后，应满足：

- 可以从首页走完整条定制流程。
- 首页明确展示服务名称、目标客户和主打风格。
- 用户可以填写或修改身材 mock 数据。
- 用户可以选择老钱风、轻奢休闲风、颜色、版型、场景。
- 页面可以展示 3 套品牌化推荐方案。
- 用户可以选择衣型、颜色、面料和细节。
- 价格会根据 mock 选项变化。
- 用户可以进入订单确认页。
- 订单确认页展示服务类型、顾问复核、服务标准和量体安排。
- 点击下单后出现成功提示。
- 页面在手机尺寸下无明显错位、遮挡、横向溢出。

## 14. 后续升级方向

简单版完成后，可以逐步升级：

- 接入微信登录。
- 接入真实用户数据。
- 接入真实订单和支付。
- 增加后台商品和面料管理。
- 增加真实 AI 推荐逻辑。
- 增加图片上传和身材识别。
- 增加顾问工作台。
- 增加会员和复购体系。

## 15. 结论

第一版建议控制在“可演示、可点击、可讲清业务逻辑”的范围内。它的核心价值不是立刻上线，而是快速验证产品体验：用户如何提供身材和审美数据，系统如何生成定制方案，用户如何完成搭配并产生购买意愿。
